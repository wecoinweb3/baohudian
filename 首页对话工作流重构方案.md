# 首页对话工作流重构方案

## 1. 背景与问题

当前首页对话流程以 `src/pages/ChatSetupPage.tsx` 中的单一 `handleSubmit` 为核心，承担了：

- 意图判断
- 参考图澄清
- Logo 预处理
- 局部微调
- 正式出图

这导致两个根本问题：

1. **理解类任务与执行类任务混在一起**
   - 例如“提取图中文字后重新设计”本质是“先分析，再让用户确认，再执行生成”，但当前会直接进入生图。
2. **缺少显式会话阶段**
   - 目前消息列表承担了过多业务语义，但没有统一表达“等待澄清 / 等待确认 / 可生成 / 生成中”等状态。

## 2. 本次重构目标（第一阶段）

第一阶段不做大爆改，只优先落地最关键主链路：

### 目标

当用户上传参考图，并明确选择：

- “提取图中的文字信息，重新制作新设计”
- 或语义等价表达

系统应执行：

1. 调用大模型识图/整理提示词
2. 返回一份 **可编辑的提示词草稿**
3. 将草稿 **回填到输入框**
4. 提示用户“请修改确认后再点击发送”
5. 用户再次发送后，才真正进入生图流程

## 3. 目标架构

### 3.1 会话阶段（Conversation Stage）

第一阶段先引入轻量状态：

```ts
type ConversationStage = 'idle' | 'awaiting_prompt_confirmation' | 'generating';
```

说明：

- `idle`：默认空闲，可正常输入并触发意图判断
- `awaiting_prompt_confirmation`：系统已经根据参考图整理出一版提示词，等待用户确认/修改
- `generating`：正式生图中

### 3.2 消息类型（Message Kind）

为 `ChatMessage` 增加轻量 `kind` 字段：

```ts
type ChatMessageKind =
  | 'text'
  | 'clarification'
  | 'prepared_prompt'
  | 'generation_progress'
  | 'generation_result'
  | 'error';
```

其中：

- `clarification`：参考图意图澄清
- `prepared_prompt`：系统已整理出提示词，等待用户确认
- `generation_progress`：生成进度
- `generation_result`：正式结果

## 4. 责任边界调整

### 4.1 后端新增 Prepare 接口

新增：

`POST /api/generate/prepare-from-reference`

职责只做：

- 看参考图
- 提取图中文字与信息层级
- 按保护垫设计场景整理为一段可编辑提示词
- 返回给前端，不直接生图

返回格式：

```json
{
  "success": true,
  "summary": "我已根据参考图提取并整理出一版设计说明，你可以先修改，再点击发送生成。",
  "preparedPrompt": "请设计一款保护垫...",
  "extractedTexts": ["标题", "电话", "安全提示"]
}
```

### 4.2 `/generate/canvas` 只负责最终生成

当用户已经确认输入内容后，前端才调用 `/generate/canvas`。

## 5. 前端工作流调整

### 5.1 新流程：提取图中文字后重做

1. 用户上传参考图
2. 用户点击建议项或输入：`提取图中的文字信息，重新制作新设计`
3. 前端识别为 `reference_extract_prepare`
4. 调用 `/api/generate/prepare-from-reference`
5. assistant 展示一条 `prepared_prompt` 类型消息
6. `setInput(preparedPrompt)` 回填输入框
7. 设置 `conversationStage = awaiting_prompt_confirmation`
8. 用户修改后再次点击发送
9. 进入正式生成

### 5.2 UI 原则

- 输入框上方显示轻提示：当前是“待确认提示词”状态
- 允许用户直接编辑
- 允许用户清空或重新上传参考图重新准备

## 6. 为什么这能从根上缓解逻辑混乱

这次不是继续堆场景分支，而是先建立一个统一原则：

> **分析类任务先产出中间结果；执行类任务必须等待用户确认后再继续。**

这样后续其它场景也可复用：

- 保留排版改字
- 只参考配色与布局
- OCR 提取后改写
- 根据上一版重写提示词

都可以先进入 `prepare` 阶段，而不是立刻生成。

## 7. 后续建议（第二阶段）

### 7.1 将 `handleSubmit` 拆成三层

- `resolveIntent()`
- `prepareTask()`
- `executeGeneration()`

### 7.2 引入统一会话上下文

```ts
interface ConversationContext {
  stage: ConversationStage;
  latestDraft?: ConversationDesignDraft;
  uploadedLogos: UploadedImage[];
  referenceImages: UploadedImage[];
  preparedPrompt?: string;
  pendingIntent?: string;
}
```

### 7.3 引入 OCR / 版面分析

若后续要提升“提取文字后重做”的稳定性，建议加入：

- OCR（如 PaddleOCR）
- 布局分析
- 再交给大模型做语义整理

## 8. 本次编码实现范围

本次实现只覆盖第一阶段：

- 新增方案文档
- 新增 `prepare-from-reference` 接口
- 前端新增“待确认提示词”阶段
- 实现“提取图中文字后回填输入框，用户确认后再生图”流程
- 不对所有其他场景做全面工作流迁移

## 9. 验收标准

满足以下条件即视为本阶段完成：

1. 用户上传参考图并选择“提取图中文字后让我修改”
2. 系统不会直接出图
3. 系统会生成一版整理好的提示词并回填输入框
4. assistant 会明确提示用户“请修改确认后再发送”
5. 用户二次点击发送后，才进入正式出图
6. 项目可正常构建通过
